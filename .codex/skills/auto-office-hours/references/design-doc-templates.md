# Design Doc Templates

Write the approved intake document to `.agent/work/<change-name>/INTAKE.md`.

The INTAKE is a faithful record of what the user approved. Use the user's language where possible. When the agent reframed something and the user accepted the reframe, capture the accepted version and note it was a reframe. Do not editorialize or add interpretation the user did not approve.

## Startup Mode Template

```markdown
# Intake: {title}

Generated by auto-office-hours on {date}
Active change: {change-name}
Status: DRAFT
Mode: Startup
Work scale: {bug / feature / capability / roadmap}
Work shape: {feature / refactor / parity / audit / migration / coverage / content / mixed}

## Objective
{the user's final refined objective — not the initial framing, not the agent's rewrite}

## Broader Intent
{the larger goal this spec serves, even if this spec addresses only part of it}

## Demand Evidence
{from Q1: specific quotes, numbers, behaviors demonstrating real demand}

## Status Quo
{from Q2: concrete current workflow users live with today}

## Target User & Narrowest Wedge
{from Q3 + Q4: the specific human and the smallest version worth paying for}

## Constraints
{from diagnostic}

## Scope Coverage
- Included: {material request items covered by this change}
- Deferred: {items outside this change and why; omit if none}
- Anti-goals: {explicit exclusions for this change}
- Needs decision: {questions or options that would change scope; omit if none}

## Rejected Framings
{directions the user explicitly ruled out during conversation, with their reasoning. Omit if none.}

## Scope Preservation
{whether this preserves the user's full stated intent or intentionally decomposes it}

## Premises
{from premise challenge}

## Approaches Considered
### Approach A: {name}
{from alternatives}
### Approach B: {name}
{from alternatives}

## Recommended Approach
{chosen approach with rationale}

## Key Assumptions and Risks
{assumptions that would change the approach if wrong; risks that affect execution}

## Deferred Scope
{ideas surfaced during discussion that exceed this spec's boundary. Recommend capturing in ROADMAP.md. Omit if nothing was deferred.}

## The Assignment
{one concrete real-world action, not "go build it"}
```

## Shape Context
{shape-specific section — include the one that matches the work shape:}
{- Feature: "What Makes This Cool" — the core delight, novelty, or "whoa" factor}
{- Parity: "Closure Target" — the reference system, gap landscape, what "closed" means}
{- Audit: "Key Questions" — what the audit must answer, what decision depends on findings}
{- Refactor: "Structural Goal" — what invariant holds, what structural problem is solved}
{- Migration: "Target State" — source state, target state, compatibility constraints}
{- Coverage: "Risk Areas" — what is undertested, what types of tests are needed}

## Rules

- Include `Supersedes:` in the header only if a prior design doc exists for this change.
- The Assignment / Next Steps section must be a specific action the user can do today: "interview 3 ops managers" not "think about your target market."
- Save to `.agent/work/<change-name>/INTAKE.md`, never to host-specific paths like `~/.gstack/`.
- The Objective must use the user's final refined language. If the objective evolved during conversation, capture the last approved version.
- Shape Context uses the section that matches the work shape. Do not include "What Makes This Cool" for parity, audit, refactor, migration, or coverage work.
