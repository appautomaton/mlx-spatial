**What "closed" means:** Per-module numerical parity probes pass against vendored MLX outputs within agreed tolerances, AND an end-to-end image-to-3DGS smoke produces a sensible file from a known input image. Matches the parity discipline used in Phase 1 (`2026-05-20-gap-matrix-parity`) and Phase 2 (`2026-05-20-production-pipeline-parity`).

## Constraints

- **Apple Silicon target only.** No Linux/CUDA inference path. xformers and flash_attn are explicitly out (consistent with upstream's MLX backend behavior).
- **Inference only.** No training, no fine-tuning, no data ingestion.
- **No weight redistribution this change.** `LICENSE_MODEL` and `LICENSE_generated_samples` from `apple/ml-lito` must be read before any local conversion; HF/`mlx-community` publish is deferred.
- **0.0.1 release boundary is sacred.** This change must not touch any file in the active 0.0.1 release commit group. All LiTo work lands in new modules, new tests, a new CLI script, new docs.
- **Agentic collab constraint.** Slices must be self-contained enough that a sub-agent (Codex helper, Claude sub-agent, or fresh-session Claude) can pick one up cold with `INTAKE.md` + `SPEC.md` + `PLAN.md` as its only context. Per-slice acceptance criteria, verification commands, and parity-probe scripts are first-class.
- **Memory budget.** DiT inference must fit M4 Max resident memory. Memory profile pattern follows `HYWORLD2_MEMORY_PROFILES` in `hyworld2_inference.py`. Exact ceiling determined at slice 0.

## Rejected Framings

- **Thin wrapper around upstream MLX code.** Rejected because pipeline shape and idioms would diverge from TRELLIS.2 / HY-World / SAM3D, harming testability and the agentic-collab handoff story.
- **Clean re-implementation, upstream as numerical oracle only.** Rejected because the upstream MLX path is already working code on Apple Silicon — ignoring it spends time re-deriving what's already proven.
- **Skinny end-to-end then deepen (approach B).** Rejected because parity discipline at every slice is a hard constraint for this project; landing silently divergent outputs first violates it.
- **Render-first risk slice (approach C).** Rejected as the default ordering, but **kept as a re-routing target** if slice 0 surfaces blockers in the LF render path.
- **Visual parity or smoke-only verification.** Rejected in favor of numerical parity probes at the tokenizer-latent and DiT-trajectory level.
- **Wait until 0.0.1 publishes to PyPI.** Rejected as idle time; release is execute-complete and LiTo won't touch its commit boundary.
- **Bundle LiTo into a 0.1.0 minor release.** Rejected as premature release coordination; LiTo will land on `main` and the release cut is a separate decision.
- **Weight redistribution this change.** Rejected pending `LICENSE_MODEL` review.

## Approaches Considered

### Approach A: Tokenizer-first depth (selected)

Slice sequence (preliminary, refined by auto-plan):
0. Vendor `apple/ml-lito`; license review; convert checkpoints to MLX safetensors; `lito_assets.py` + validators; slice-zero scan of upstream MLX path coverage.
1. Tokenizer port → `lito_tokenizer.py` + per-tensor parity probes.
2. DiT generator port → `lito_dit.py` + denoising-trajectory parity probes.
3. LF-conditioned GS render → `lito_render.py` + smoke render.
4. `lito.py` CLI + `mlx-spatial-lito` script + end-to-end image-to-GS smoke.
5. `docs/lito.md`, ROADMAP Phase 3 entry, architecture map update.

Effort: comparable to Phase 2 production-pipeline-parity scale. Risk: lowest per-slice. Pros: matches established parity discipline; tokenizer is the most novel architectural piece so landing it first unlocks downstream confidence; per-slice agentic-collab handoffs are clean. Cons: first end-to-end output arrives at slice 4.

### Approach B: Skinny end-to-end then deepen

Land a minimal end-to-end pipeline (smoke only, no parity) in slice 1, then retrofit numerical parity slice-by-slice. Effort: faster to first smoke. Risk: silently divergent outputs may survive multiple slices; weakens the agentic-collab story because slice 1 touches every module at once.

### Approach C: Render-first risk slice

LF-conditioned render isolated (driven by ground-truth latents) in slice 1, then tokenizer, DiT, CLI. Effort: comparable. Risk: front-loads research risk — appropriate **only if** slice 0 surfaces blockers. Upstream's MLX backend already proves render works, so the apparent research risk is actually low. Kept as a re-routing target.

## Recommended Approach

**A — Tokenizer-first depth.** Decision basis:
- Mlx-spatial Phase 1 and Phase 2 closed gaps module by module with parity probes; reviewers, contributors, and sub-agents are calibrated to that shape.
- Tokenizer is the most architecturally novel piece (8192 × 32 point-cloud latents are unlike O-Voxel or ShapeVAE) — landing it first unlocks everything downstream.
- Upstream's MLX backend already proves end-to-end feasibility on Apple Silicon, so an early skinny smoke (B) isn't needed to de-risk.
- Per-slice acceptance criteria, isolated modules, and parity probes are exactly the shape needed for clean agentic-collab handoff.

**What would change the recommendation to C:** Slice 0's scan of `vendors/ml-lito/plibs/` reveals that LF-conditioned rendering requires Metal/MLX ops not yet in mlx-spatial. Slice 0 is the explicit re-routing checkpoint.

## Key Assumptions and Risks

- **Risk:** LF-conditioned GS render does not compose with existing GS infra. **Mitigation:** slice-zero scan; re-route to approach C if confirmed.
- **Risk:** Upstream MLX path covers only parts of the pipeline. **Mitigation:** slice-zero coverage audit; slices that lack MLX reference grow to include a from-PyTorch port with stronger numerical-oracle dependency.
- **Risk:** `LICENSE_MODEL` forbids even local conversion. **Mitigation:** read the license first thing in slice 0; if conversion is blocked, switch to runtime `.ckpt` loading at the cost of MLX-native performance.
- **Risk:** DiT exceeds M4 Max resident memory. **Mitigation:** memory profile system from `HYWORLD2_MEMORY_PROFILES`; tile or stream layers as needed.
- **Assumption (agentic collab):** Sub-agents can pick up a slice cold with `INTAKE.md` + `SPEC.md` + `PLAN.md` as full context. Validated by writing slice acceptance criteria in self-contained form and including explicit verification commands per slice.
