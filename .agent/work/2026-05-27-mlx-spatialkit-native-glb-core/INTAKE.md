## Objective
Build `mlx-spatialkit` as a native C++ / Metal 4 package that fulfills `mlx-spatial`'s GLB generation needs, starting with Pixal3D decoded `shape_decoder_fields.npz` + `texture_decoder_pbr.npz` -> textured `model.glb`.

## Broader Intent
Give `mlx-spatial` a clean, performant, memory-safe native export backend for Apple Silicon 3D generation outputs while keeping Python as thin orchestration and keeping heavy generated artifacts out of the repo.

## What Makes This Cool
This turns Pixal3D's expensive post-model export stage into a reusable native spatial toolkit: mesh extraction, mesh diagnostics, cleanup/simplification, texture baking, and GLB export become a package boundary that future model pipelines can call.

## Structural Goal
Create `packages/mlx-spatialkit` as a separate Python distribution named `mlx-spatialkit` with import package `mlx_spatialkit`. The hot path must live in native C++ / Objective-C++ / Metal 4 code; Python APIs only load inputs, validate contracts, and call native bindings.

## Closure Target
Reference system: current `mlx-spatial` Pixal3D export code plus the organized local fixtures under `inputs/mlx-spatialkit`.

Closed means `mlx-spatialkit` can take Pixal3D decoded NPZ artifacts and produce a textured GLB through native-owned export stages with clear diagnostics, tests, and integration points for `mlx-spatial`.

## Risk Areas
- Native packaging and Metal toolchain setup can fail before implementation quality is testable.
- Pixal3D mesh topology is large and messy: the known reference path starts from about 8.3M faces and exports around 212k faces even with a 50k target.
- A full remeshing engine is too broad for this change; the first native core must make GLB export work without pretending to be upstream `cumesh` parity.
- Current Python export path uses NumPy, `fast_simplification`, SciPy KD-tree, and Python `xatlas`; the new package must replace or isolate those hot loops behind native APIs.

## Constraints
- Package path: `packages/mlx-spatialkit`.
- Distribution name: `mlx-spatialkit`; import name: `mlx_spatialkit`.
- Do not require MLX in `mlx-spatialkit`; accept NumPy/buffer-compatible arrays and files.
- Use C++ / Objective-C++ / Metal 4 for hot paths.
- Python must not own mesh extraction, mesh cleanup/simplification loops, texture raster/sampling loops, or GLB binary assembly loops.
- Use `/tmp` for heavy generated outputs during development and tests unless a small fixture is intentionally committed.
- Keep `inputs/`, `outputs/`, `vendors/`, and `weights/` out of package artifacts.
- Follow the Automaton lifecycle: frame -> plan -> execute -> verify.

## Scope Coverage
- Included: native package frame, native mesh contracts, Pixal3D decoded-NPZ input contract, FlexiDualGrid mesh extraction target, diagnostics, cleanup/simplification interface, Metal 4 texture bake path, GLB export API, `mlx-spatial` integration boundary, unit/integration tests, and repo-clean artifact rules.
- Deferred: full general remeshing engine, complete upstream `cumesh` parity, replacing `xatlas` completely if a wrapped native UV interface is sufficient for first GLB success, broad geometry toolkit features, and support below Metal 4.
- Anti-goals: Python-first implementation, pure diagnostic-only package, MLX dependency inside `mlx-spatialkit`, runtime Torch dependency, generated-output pollution in the repo, and claiming full remesh parity before it exists.

## Scope Preservation
This preserves the user's full stated intent as one coherent capability spec, not a roadmap. Later work may deepen remeshing or replace UV dependencies, but this change is scoped to the first native GLB core that `mlx-spatial` can actually use.

## Recommended Approach
Approach B2: Native GLB Fulfillment Core, because it keeps the performance-critical architecture native from day one while limiting the first deliverable to Pixal3D decoded artifacts -> working textured GLB.

## Key Assumptions and Risks
- "meta4" is treated as "Metal 4"; the current machine has `xcrun metal -v` available.
- `inputs/mlx-spatialkit/pixal3d-1024-cascade-decoded-pbr` is the primary real fixture for decoded Pixal3D model outputs.
- `inputs/mlx-spatialkit/pixal3d-1024-cascade-glb-reference/model.glb` is a visual/output reference, not a byte-for-byte parity target.
- A native UV path may initially wrap the C++ `xatlas` library behind a replaceable interface; replacing xatlas entirely is deferred unless planning proves it is required for first success.

## Rejected or Deferred
- A Python-first or stub-heavy package was rejected because it hides the performance problem.
- A broad roadmap was rejected because the target is one coherent capability: native GLB generation for `mlx-spatial`.
- Full remeshing parity was deferred because the first requirement is to make the Pixal3D export path work reliably and cleanly.
