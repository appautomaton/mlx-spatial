## Objective

Close all remaining inference gaps across SAM 3D, TRELLIS.2, and HY-World 2.0 — consolidating the three pending roadmap phases (MOT/GS Rendering, Texturing/Sparse Interpolation, Metal Alternatives/Postprocessing) into one spec. 10 gaps remain after removing already-ported modules (SAM-MOT, SAM-SHORTCUT) and training-only items (HW-FRUSTUM).

## Broader Intent

mlx-spatial aims to be a complete MLX-native inference library for 3D spatial models on Apple Silicon. This spec closes the final inference parity gap so the library delivers full production output across all three pipelines with Mac-native alternatives for every CUDA-dependent feature.

## Shape Context

### Parity: Closure Target

Reference systems are the three vendor implementations (`vendors/TRELLIS.2`, `vendors/sam-3d-objects`, `vendors/HY-World-2.0`). "Closed" means numerically matching output against the PyTorch reference within defined tolerance, verified by a dedicated parity test. Where numeric parity is impractical (e.g., `mesh_to_flexible_dual_grid` which replaces a native C++/CUDA library), the bar is performance-comparable, correct-looking output.

### Migration: Target State

Source: vendor CUDA-dependent operations (o_voxel, cumesh, nvdiffrast, flex_gemm, pytorch3d, open3d, igraph). Target: pure Python/NumPy/MLX/Metal alternatives running on Apple Silicon with no CUDA runtime dependency.

## Constraints

- No runtime PyTorch, CUDA, or vendor-native library dependencies
- All gaps must produce correct output on Apple Silicon (M1 baseline for Metal compute)
- Existing test patterns (`test_*_parity.py`) are sufficient for verification
- The Metal GS rasterizer in `gs_rasterize.py` is the foundation — gaps extend it, not replace it
- Performance optimization and CI/release infrastructure are intentionally excluded

- **MOT was listed as a P1 gap but is already ported:** Verified by reading `sam3d_ss_flow.py` — full MOT transformer with shape-protected self-attention, cross-attention, and adaLN modulation. Two parity tests exist. Removed from scope.

## Approaches Considered

### Approach A: Dependency-Ordered (bottom-up)

**Summary:** Close gaps in dependency order — independent utilities first (GS/SH), then TRELLIS.2 texturing chain, then mesh postprocessing. Mirrors Phase 1's gap matrix pattern.
**Effort:** L
**Risk:** Low
**Pros:** No rework from shared infrastructure. Proven pattern from Phase 1. Three groups are independently executable by parallel subagents.
**Cons:** No pipeline-level completion signal until late. User-visible progress comes last.
**Reuses:** Gap matrix dependency graph, existing test patterns, existing GS renderer

### Approach B: Pipeline-Complete (per-pipeline)

**Summary:** Close all gaps for one pipeline at a time — TRELLIS.2 first, then SAM 3D, then HY-World cleanup.
**Effort:** L
**Risk:** Medium
**Pros:** Each pipeline hits completion independently — clear progress gates. TRELLIS.2 texturing ships early.
**Cons:** Shared GS work risks duplication. SAM 3D postprocessing needs GS rendering that TRELLIS.2 texturing doesn't.
**Reuses:** Existing per-pipeline module patterns, per-pipeline test suites

### Approach C: Impact-Sorted (visible features first)

**Summary:** Close gaps by user-visible impact — texturing first, then MOT (already done), then quality improvements, then remaining utilities.
**Effort:** L
**Risk:** Medium
**Pros:** Ships flashiest results fastest. Texturing is the only wholly new output mode.
**Cons:** GS infrastructure built ad-hoc for texturing may need restructuring for SAM 3D.
**Reuses:** Existing renderer, existing pipeline entry points

## Recommended Approach

**Approach A — Dependency-Ordered.** Selected by the user. Mirrors the Phase 1 gap matrix pattern. The three groups are naturally independent and can be executed in parallel by subagents. The GS renderer already provides the shared foundation, so the "shared first" risk is low.

## Key Assumptions and Risks

- **mesh_to_flexible_dual_grid is the highest-risk gap.** The vendor uses native `o_voxel` C++/CUDA. The MLX codebase has `flexible_dual_grid_to_mesh_np()` (the reverse direction) which proves the FDG topology is understood. A forward mesh→FDG conversion requires voxelizing a triangle mesh — feasible in pure Python/NumPy but may need optimization.
- **SAM-LAYOUT is the largest single gap** (1295 lines of vendor code depending on PyTorch3D/Open3D/scipy). Replacing with Mac-native alternatives (NumPy ICP, Metal render-and-compare) is architecturally significant.
- **SAM-HOLES multi-view mincut** depends on igraph (C library). A pure Python graph mincut implementation or an alternative hole-filling strategy may be needed.
- **Dual-contouring remeshing** (TR-REGRID) requires a Mac-native alternative to `cumesh.remeshing`. May be scoped to "best available alternative" rather than numeric parity.
- **SH degree 4** is near-trivial — constants exist, just need to add the evaluation terms.
- Three groups can be executed in parallel by subagents with shared knowledge of the existing codebase.
